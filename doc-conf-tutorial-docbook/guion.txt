<h2>Trabajando: definición de entidades y descomposición en varios ficheros fuente
<p>
Las entidades tienen varios usos diferentes. Los más corrientes son:

<dl>
    <dt>Codificación de caracteres
    <dd>sirven para codificar, en ascii, caracteres nacionales o de difícil representación.
    <dt>Entidades generales internas
    <dd>sirven para crear acrónimos o macros.
    <dt>Entidades generales externas
    <dd>se usan para partir el código fuente en varios ficheros.
</dl>

Encontrarán información más detallada en <a href="file:///mnt/taca3/docbook/tdg/1.0.3/en/html/ch01.html#S-ENTITIES">The Definitive Guide</a>.

<h2>Trabajando: equivalencias prácticas entre Docbook SGML y Docbook XML
<p>
<ul>
	<li>Con alguna rara excepción ambas DTD son iguales.
	<li>Como XML es compatible con SGML es posible usar el ciclo de producción SGML sin inconveniente (la vía DSSSL).
	
</ul>

<h2>Trabajando: diferencias prácticas entre Docbook SGML y Docbook XML
<p>

<ul>
	<li>El sufijo :-)
	<li>El código XML permitiría usar las herramientas XSL (no estudiadas en este curso).
	<li>Con XML los elementos (las marcas) siempre estarán en minúsculas.
	<li>Con XML siempre se escribirá el contenido de los atributos entre comillas.
	<li>Con XML las marcas que no necesitan ser cerradas se escriben &lt;marca/>.
	<li>Con XML la primera línea siempre será una declaración XML explicando la versión de XML y el juego de caracteres.
</ul>

<h2>Trabajando: personalización de DTD
<p>

SGML y XML están concebidos para poder hacer modificaciones o extensiones a las DTD.<p>
En la práctica esta necesidad es muy rara. Además, al modificar la DTD perdemos facilidades de portabilidad entre sistemas y exige la ampliación o modificación de las hojas de estilo.<p>
Por todo ello no se considerará en este curso.

<h2>Trabajando: personalización de hojas de estilo DSSSL
<p>

Es posible personalizar el comportamiento de las hojas de estilo sin modificarlas usando un <a href="ldp.dsl">«driver»</a> DSSSL.<p>
Este  «driver» sirve para <em>matizar</em> a las hojas actuales y deben incluir una referencia a las mismas para que el procesador DSSSL las localice.<p>
El driver se invocaría, por ejemplo: <p><center>docbook2pdf -d ldp.dsl#print fichero.xml</center>


<h2>Trabajando: personalización de hojas de estilo XSL
<p>

Dado que el ciclo de producción XSL no está tan maduro como el DSSSL no consideraremos en este curso la personalización de sus hojas de estilo.

<h2>Complicaciones más comunes: errata Debian/Linex
<p>
 
 Una errata en las hojas de estilo de la actual versión de Linex impide que algunas imágenes se incluyan en los ficheros pdf de salida. Esto puede resolverse fácilmente usando un «driver», como por ejemplo: 
 <a href="http://tldp.org/authors/tools/ldp.dsl">http://tldp.org/authors/tools/ldp.dsl</a>. El modo de uso sería:<p>
 <center>docbook2pdf -d ldp.dsl#print fichero.xml</center>
 
 

<h2>Complicaciones más comunes: juegos de caracteres
<p>
Hasta ahora el juego de caracteres predeterminado en Linux para el español ha sido Latin1 (iso-8859-1). La nueva versión adaptada al Euro es Latin9 (iso-8859-15). <br>
Ambos juegos son muy parecidos. Como el sistema funciona siempre funciona con uno y otro no es fácil que se puedan mezclar.<p>
Sin embargo el juego predeterminado en la tecnología XML es UTF-8 y es probable que nos encontremos muchos ficheros así codificados. A la hora de modificar un fichero XML hay que tener presente que la codificación del fichero y la que usa la aplicación coincidan. Si no es así se pueden hacer transformaciones con programas como el analizador xmllint o <strong>recode</strong>, un poderoso conversor de codificaciones.

<h2>Complicaciones más comunes: usando ilustraciones en PS/PDF
<p>

Como hemos visto TeX es el motor de composición para los formatos de reproducción «impresa». <p>
Hay dos motores TeX, el clásico que crea ficheros PostScript (y el intermedio DVI) y PDFTeX que crea PDF.<br>
Curiosamente ambos motores no manejan los mismos formatos gráficos para ilustraciones. El lío aumenta al tener en cuenta la salida vía web.<p>
Hay varias formas de sortear este problema pero la más sencilla es optar siempre por la salida PDF y HTML y utilizar los formatos gráficos PNG y JPG. Y no especificar nunca el atributo <strong>format</strong>. 

<h2>Complicaciones más comunes: usando ilustraciones en HTML
<p>

<dl>
<dt>Ilustraciones<dd>Algunas versiones de docbook-utils varían de comportamiento al crear documentos HTML descompuestos en varios ficheros. Así en unos casos todos los ficheros html se crearán en el directorio actual y en otros lo hará en un subdirectorio. En este segundo caso será necesario copiar las imágenes a una ruta relativa igual a la que se introdujo en el código fuente para su correcta visualización en el navegador.
<dt>Adornos de estilo<dd>Según la configuración de las hojas de estilo éstas podrán podrán usar adornos con forma de fichero gráfico. Si usa esta opción deberá copiar a mano estas imágenes (en mi caso en /usr/share/sgml/docbook/dsssl-stylesheets-1.74b/images/) al subdirectorio adecuado.
</dl>

<h2>Editores estructurados
<p>

El código Docbook es bastante prolijo. Puede que prefiera usar un editor estructurado más avanzado y cómodo. Algunas alternativas libres o gratis:
<dl>
	<dt>vim
	<dd>dispone de un modo de coloreado para XML, pero no asiste a la escritura.
	<dt>jedit
	<dd>Es un editor java, portable, con modos de edición de XML, XSLT, etc.
	<dt>emacs+psgml
	<dd>Es un muy poderoso editor estructurado. Una versión más acabada de psgml es psgmlx.
	<dt>xxe
	<dd>Es un editor estructurado en java.
</dl>
